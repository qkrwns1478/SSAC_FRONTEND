# Token Optimization Protocol (FE)

## 트리거 조건

- 작업 시작 전 (자동 — 매 작업)
- 사용자가 "느려", "토큰", "최적화" 언급 시 (자동)
- harness-audit.md 실행 결과 이상 감지 시 (자동)
- 스프린트 종료 시점 (수동)

## 실행 순서

STEP 1. 컨텍스트 최소화
STEP 2. 미사용 파일 감사
STEP 3. 프로토콜 중복 제거
STEP 4. 최적화 결과 기록

---

## STEP 1. 컨텍스트 최소화

### 작업 시작 전 필수 점검

□ 현재 작업과 직접 관련된 파일만 로드한다
□ 아래 파일은 명시적 요청 없이 로드하지 않는다

로드 금지 파일 목록 (작업 무관 시):

- src/mocks/\*\* (테스트 작업 외)
- docs/debug-log.md (진단 작업 외)
- .next/\*\* (빌드 산출물)
- node_modules/\*\* (의존성 파일)
- coverage/\*\* (커버리지 산출물)
- src/\*_/_.test.tsx (테스트 작업 외)

### 파일 로드 우선순위

1순위 : 현재 작업 SC와 직접 연관된 컴포넌트/훅/페이지
2순위 : 해당 파일이 import하는 파일
3순위 : api-contract/generated/api-types.ts (API 연동 시)
4순위 : 프로토콜 파일 (트리거 조건 충족 시)

→ 4순위 이하 파일은 로드하지 않는다

## STEP 2. 미사용 파일 감사

### 즉시 삭제 대상

□ 아무 곳에서도 import되지 않는 컴포넌트 파일
→ 검증: grep -r "import._{컴포넌트명}" src/
□ 대응하는 컴포넌트가 없는 테스트 파일
□ 빈 MSW 핸들러 파일 (핸들러 배열이 비어있는 파일)
□ 최근 30일 이상 수정되지 않은 임시 파일
→ 패턴: _.tmp, _\_backup._, _\_old._
□ 주석 처리된 코드 블록이 30줄 이상인 파일

### 검토 후 삭제 대상

□ TODO 주석이 90일 이상 방치된 파일
□ 사용되지 않는 타입 정의 (src/types/index.ts 내)
□ 중복된 유틸리티 함수 (동일 기능 함수가 2개 이상)

### 삭제 전 확인 절차

1. git log --oneline -5 {파일명} 으로 최근 커밋 확인
2. grep -r "{컴포넌트명}" src/ 으로 참조 여부 확인
3. 참조 없음 확인 후 삭제
4. 삭제 후 빌드 성공 확인 (npm run build)
5. 관련 테스트 통과 확인 (npm run test)

출력 형식:

```
[미사용 파일 감사 결과]
즉시 삭제 : 2개
- src/components/OldButton.tsx (미참조 45일)
- src/mocks/handlers/unused.handler.ts (빈 핸들러)
검토 필요 : 1개
- src/types/index.ts 내 LegacyUserType (미사용 타입)
```

## STEP 3. 프로토콜 중복 제거

□ 동일한 체크리스트 항목이 2개 이상의 프로토콜에 존재하는가
→ 공통 항목을 CLAUDE.md 전역 규칙으로 이동
→ 개별 프로토콜에서 해당 항목 삭제

□ 프로토콜 파일이 200줄을 초과하는가
→ 핵심 규칙만 남기고 예시를 별도 파일로 분리
→ 예시: docs/examples/{프로토콜명}-examples.md

□ 트리거 조건이 중복되는 프로토콜이 있는가
→ 중복 트리거를 가진 프로토콜 병합 검토

프로토콜 파일 적정 크기 기준:

- 핵심 규칙 파일 : 200줄 이하
- 예시 포함 파일 : 400줄 이하
- 400줄 초과 시 : 분리 필수

출력 형식:

```
[프로토콜 중복 감사 결과]
중복 항목 발견 : N개
- api-contract 확인 체크 (new-feature.md / sc-harness.md)
분리 필요 파일 : N개
- testing.md (현재 450줄 → 예시 분리 권장)
```

- 즉시 삭제 : N개
- 검토 후 삭제 : N개

### 프로토콜 중복 제거

- 제거된 중복 항목 : N개
- 분리된 예시 파일 : N개

→ 예상 토큰 절감율 : N%
